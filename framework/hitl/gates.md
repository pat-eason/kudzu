# hitl/gates.md — Human-in-the-Loop Gate Definitions

Gates are mandatory checkpoints where the pipeline pauses for human review.
No agent may proceed past a gate without a valid GATE_[N]_DECISION.md file.

A gate decision file is created by the human — not generated by an agent.
Agents may surface the information needed for a decision, but cannot make it.

---

## GATE 1: Discovery Review

**Triggered after:** Project Planner produces RESEARCH_SYNTHESIS.md + VIABILITY_REPORT.md
**Blocks:** PRD writing

**What the human reviews:**
- RESEARCH_SYNTHESIS.md — synthesized pro/con research findings
- VIABILITY_REPORT.md — Concept Reviewer's implementation viability assessment
- PRO_FINDINGS.md and CON_FINDINGS.md — raw researcher outputs if desired

**Decisions available:**
```
APPROVED        → proceed to PRD writing
RESEARCH_AGAIN  → re-run research with amended direction (provide feedback)
REDIRECT        → change the core concept (provide new direction)
ABANDON         → stop here, no PRD
```

**File to create:** `GATE_1_DECISION.md`
```markdown
# Gate 1 Decision

DECISION: [APPROVED | RESEARCH_AGAIN | REDIRECT | ABANDON]
DATE: [date]
FEEDBACK: |
  [Required if RESEARCH_AGAIN or REDIRECT]
  [Optional if APPROVED — will be incorporated into PRD]
  [N/A if ABANDON]
```

---

## GATE 2: PRD Approval

**Triggered after:** Project Planner produces PRD.md
**Blocks:** Architecture + chunk decomposition

**What the human reviews:**
- PRD.md — full product requirements document
- GATE_1_DECISION.md — verify feedback was addressed

**Decisions available:**
```
APPROVED        → proceed to architecture
REVISE          → revise PRD with feedback (provide specific feedback)
```

**File to create:** `GATE_2_DECISION.md`
```markdown
# Gate 2 Decision

DECISION: [APPROVED | REVISE]
DATE: [date]
FEEDBACK: |
  [Required if REVISE — be specific about what sections need changing]
  [Optional if APPROVED — will be incorporated into architecture notes]
APPROVED_VERSION: [PRD version hash or "v1" if first pass]
```

---

## GATE 3: Architecture Approval

**Triggered after:** Software Architect produces CHUNKS.json + IMPL_SPECS/ + REVIEW_SPECS/
AND Concept Reviewer produces DECOMP_REVIEW.md

**Blocks:** Linear project creation + implementation start

**What the human reviews:**
- CHUNKS.json — full implementation plan
- DECOMP_REVIEW.md — Concept Reviewer's assessment of the decomposition
- IMPL_SPECS/ — spot-check implementation specs for 2-3 chunks
- ADVERSARY_REPORT.md if present

**Decisions available:**
```
APPROVED        → proceed to Linear project creation + implementation
REVISE_CHUNKS   → revise decomposition (provide feedback)
REVISE_PRD      → the PRD needs updating before architecture can proceed
                  (returns to Gate 2 flow)
```

**File to create:** `GATE_3_DECISION.md`
```markdown
# Gate 3 Decision

DECISION: [APPROVED | REVISE_CHUNKS | REVISE_PRD]
DATE: [date]
FEEDBACK: |
  [Required if REVISE — cite specific chunk ids or sections]
IMPLEMENTATION_NOTES: |
  [Optional — additional context for the engineering phase]
```

## GATE 4: Chunk Review (conditional)

**Triggered IF ANY of the following are true:**
- code-reviewer verdict is FAIL
- security-analyst verdict is FAIL or NEEDS_REVIEW
- qa-analyst coverage assessment is INSUFFICIENT
- Any agent sets `needs_human: true` in their output
- This is chunk 1 (first implementation — always human-reviewed)
- The chunk touches a system marked `load_bearing: true` in CHUNKS.json

**Does NOT trigger if:**
- code-reviewer is PASS or PASS_WITH_NOTES
- security-analyst is PASS
- qa-analyst is SUFFICIENT
- No agent flags needs_human
- Not chunk 1 or a load-bearing system

**What the human reviews:**
- REVIEW.md — code review findings
- SECURITY_REVIEW.md — if security was escalated
- TEST_REPORT.md — QA findings
- DELTA.md — what changed
- The actual diff / changed files

**Decisions available:**
```
APPROVED        → proceed to documentation + delivery
APPROVED_WITH_NOTES → proceed, but create follow-up Linear issues for notes
FIX_AND_RETRY   → fix specific issues, re-run review (no re-implementation)
REIMPLMENT      → discard implementation, re-run from engineer with guidance
ESCALATE        → bring in architect before proceeding
```

**File to create:** `GATE_4_DECISION_chunk[N].md`
```markdown
# Gate 4 Decision — Chunk [N]

DECISION: [APPROVED | APPROVED_WITH_NOTES | FIX_AND_RETRY | REIMPLEMENT | ESCALATE]
DATE: [date]
CHUNK_ID: [N]
FEEDBACK: |
  [Required if FIX_AND_RETRY — list specific issues to fix]
  [Required if REIMPLEMENT — explain what was wrong with the approach]
  [Required if ESCALATE — describe the concern]
FOLLOW_UP_ISSUES: |
  [If APPROVED_WITH_NOTES — list issues to create in Linear]
```

## Gate State Check

Agents check gate state by looking for GATE_[N]_DECISION.md.
If the file exists and DECISION is not ABANDON: read the feedback and proceed.
If the file does not exist: output a GATE_PENDING.md and stop.

GATE_PENDING.md format (agent-generated):
```markdown
# Gate Pending — Gate [N]

The pipeline has reached Gate [N] and requires human review before proceeding.

## What Needs Review
[List the files the human should read]

## Decision File to Create
Create `GATE_[N]_DECISION.md` using the template in hitl/gates.md.

## Context
[One paragraph summary of what was produced and any concerns worth flagging]
```
